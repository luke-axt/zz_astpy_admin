# 公网服务架构选型建议

## 一、选型原则

基于当前团队规模、技术储备、招聘现实及业务需求，公网服务选型遵循以下原则：

1. **团队匹配优先**：应用开发组（1 名中级 Java + 1 名初级前端）是公网服务的主要建设力量。客服模块的技术栈应当与应用开发组的能力匹配，降低招聘与协作成本。
2. **招聘现实优先**：在深圳跨境电商行业、给定薪酬区间（初级 7-12k、中级 13-17k、高级 18k+）下，Java 后端人才储备远大于 Python Web，是唯一能务实推进的选项。
3. **内网先行验证**：客服模块先在内网开发、部署和验证，待功能成熟、运行稳定后再发布到公网。降低公网直接上线的试错成本。
4. **学习成本极低**：摒弃需要额外学习复杂配置、新概念模型或新运维体系的组件。
5. **轻量化**：单台或两台低配比云服务器即可承载全部公网服务，降低硬件与运维成本。
6. **平滑兼容**：保留现有公网 FlaskAPI 服务，客服模块作为独立系统渐进式接入，而非推倒重来。

---

## 二、推荐架构：Nginx + RuoYi + Redis + PostgreSQL + Python AI

采用 **Nginx + RuoYi（Spring Boot）+ Redis + PostgreSQL**，以 **Docker Compose** 统一编排部署。AI 模块（机器人、知识库、意图识别）独立为 Python 服务，通过 HTTP/MQ 与 Java 后端通信。

### 2.2 内网开发与公网发布的阶段划分

客服模块采用**"内网开发 → 内网验证 → 公网发布"**的三阶段策略：

**阶段一：内网开发与测试**
- 在内网服务器部署 RuoYi Java + RuoYi Vue3 + PostgreSQL + Redis
- 开发 eBay 客服业务功能（消息收发、人工回复、机器人策略、知识库）
- 与现有内网系统（MySQL、若依）进行接口对接和数据联通测试
- **注意**：内网环境仅用于开发和测试，不承载 7×24 生产流量（内网无 UPS，断电风险高）

**阶段二：内网试运行验证**
- 在内网环境模拟真实客服场景，验证消息实时性、机器人回复准确率、WebSocket 稳定性
- 小范围邀请内部员工作为"买家"进行端到端测试
- 修复问题、优化性能，直至系统稳定

**阶段三：公网发布**
- 将验证成熟的客服模块部署到公网云服务器
- 保留现有公网 Python API（FlaskAPI）不变，与 Java 客服模块共存
- Nginx 统一代理：公网流量按域名/路径路由到 Python API（存量）或 Java 客服（新增）
- 客服模块正式接收真实 eBay 买家消息，开始 7×24 在线服务

### 2.3 核心组件选型说明

| 组件 | 选型 | 角色 | 推荐理由 |
|------|------|------|----------|
| **Web / 代理** | **Nginx** | 统一流量入口 | 成熟稳定，配置语法简单，直接解决「公网域名代理」和 SSL 证书管理问题。可将不同域名/路径路由到不同后端（Python API / Java 客服），天然适合多模块共存。 |
| **API 服务 / 管理后台** | **RuoYi（Spring Boot）** | 客服业务 API 与后台基础功能 | 应用开发组已熟悉若依（内网在用），用户管理、角色权限、菜单管理、字典管理、操作日志、代码生成等基础设施**开箱即用**，省去 2-3 周前端开发量。Spring Boot 单体足够支撑当前业务，无需 Spring Cloud。 |
| **前端** | **RuoYi Vue3** | 客服工作台等 Web 应用 | 若依自带 Vue3 + Element Plus 前端，与后端强配套。客服业务页面（消息列表、聊天窗口、机器人配置、知识库）在若依框架上二开即可。 |
| **AI 服务** | **Python（FastAPI）** | 机器人策略、知识库检索、意图识别 | AI 领域 Python 是绝对主流（PyTorch、LangChain、OpenAI SDK）。独立部署为微服务，通过 REST API 或 Redis 队列与 Java 后端通信，职责清晰。 |
| **异步任务** | **Spring @Async / 线程池** | 轻量异步消费 | eBay API 调用、消息解析等耗时逻辑，在 Java 侧用 Spring 异步线程池处理即可。当前业务量无需引入 RabbitMQ/RocketMQ，降低运维复杂度。 |
| **缓存 / 状态 / 轻量队列** | **Redis** | 缓存、Token 暂存、Java-Python 通信桥梁 | 一物多用：Java 后端缓存高频 Token 和会话上下文；Python AI 服务通过 Redis 获取任务或回写结果；减少数据库读写压力。 |
| **数据库** | **PostgreSQL** | 持久化 | 日均 3500 笔订单 + 多客服并发，SQLite 写串行化已成为瓶颈。PostgreSQL 行级锁、JSONB 字段、全文搜索等特性与 Java/Python 生态均匹配，且为 SaaS 多租户预留扩展空间。 |
| **进程守护** | **Docker Compose** | 7×24 进程管理 | 将 Nginx、RuoYi Java、Python AI、PostgreSQL、Redis 全部容器化编排。用 Docker 的 `restart: always` 做进程守护，环境一致性远优于裸机部署。 |

### 2.4 需求映射

| PRD 需求 | 实现方式 |
|----------|----------|
| **刷新 OC 系统 Token** | **保留现有公网 FlaskAPI**，由 Nginx 统一反向代理；Redis 缓存 Token 避免频繁刷新。长期共存，无需迁移。 |
| **独立站后端接口转发** | Nginx 将独立站域名请求转发至现有公网 FlaskAPI；FlaskAPI 再对内做鉴权、聚合、协议转换。长期共存，无需迁移。 |
| **公网域名代理（企业微信对接）** | Nginx 监听企业微信回调域名，路由到现有 FlaskAPI 对应接口。如需新功能，再评估是否迁移至 RuoYi Java。 |
| **eBay 智能客服** | **开发阶段**：在内网环境部署 RuoYi Java + Vue3，开发并验证客服业务功能。<br>**发布阶段**：成熟后部署至公网云服务器，Nginx 将客服域名流量路由至 Java 客服模块。<br>**前端**：基于 RuoYi Vue3 搭建客服工作台（消息列表、对话窗口、客户资料、人工转接、快捷回复、店铺权限等）。若依已内置用户/角色/菜单/按钮权限，前端只需专注业务模块。<br>**Java 后端**：RuoYi 提供客服 Webhook 接收接口 + RESTful API（会话管理、人工回复、店铺权限控制）；新消息与状态变更通过 **WebSocket** 实时推送至客服后台。<br>**AI 后端**：Python FastAPI 独立服务，负责机器人策略配置、知识库检索、意图识别、自动回复生成。Java 后端通过 HTTP/Redis 调用 AI 服务。<br>**存储**：Redis 暂存会话上下文；PostgreSQL 记录聊天记录与客服操作日志。 |
| **定时任务调度** | 公网 7×24 的定时需求（如 Token 刷新兜底）继续使用 APScheduler（Flask 侧），长期不变。客服模块的定时任务（如知识库同步）在 RuoYi 中用 Quartz 管理。 |

## 三、为什么不选其他方案

### 3.1 不选 Python / FastAPI 作为客服主栈

- **招聘现实**：在深圳跨境电商行业、给定薪酬区间（中级 13-17k）下，Python Web 后端人才极少。会 Python Web 的人通常去 AI/数据公司拿更高薪，不会选择跨境电商行业。
- **团队结构**：应用开发组现有 1 名中级 Java + 1 名初级前端，客服模块若用 Python，等于让 Java 开发闲置，而数据开发组的 3 名初级 Python 又无法支撑复杂业务系统的开发。
- **基础设施成本**：若用 FastAPI，用户管理、角色权限、菜单管理、操作日志等基础设施需要从零开发或基于 Admin 模板二开。而 RuoYi 已完整内置这些功能，开发速度更快。

### 3.2 不选 Spring Cloud

- **过度设计**：当前客服模块仅是新增业务系统，日均 3500 笔订单。Spring Cloud（Gateway、Nacos、Sentinel、Feign 等）属于微服务架构，适合大公司多团队协作、服务数量几十上百的场景。当前阶段用 Spring Boot 单体足够，引入微服务是自找麻烦。
- **人力不匹配**：应用开发组目前仅 1 名中级 Java，即使招到 AI 开发也才 2 人，无法支撑微服务的运维复杂度。

### 3.3 不选 Celery / RabbitMQ / Kafka

- 当前业务量下，Java 侧用 Spring 异步线程池 + Redis List 即可满足 eBay 消息的异步处理。引入独立消息中间件会增加运维负担，且应用开发组对 MQ 的熟悉度有限。
- 若未来业务增长至需要可靠消息投递（如订单状态变更、多系统联动），再引入 RabbitMQ 不迟。

### 3.4 不选 Kubernetes / Docker Swarm

- 当前阶段仅需要在一到两台云服务器上运行不到 10 个容器，引入 K8s 的运维复杂度远大于收益。Docker Compose 在单机上已实现环境一致性和进程守护，问题定位更直接。

### 3.5 不选 Jinja2 模板引擎或纯静态页面

- eBay 客服存在大量动态交互（实时消息、人工转接、表单提交、状态管理），传统服务端模板渲染开发效率低、用户体验差，无法满足「人工介入」类复杂业务场景。
- 前后端分离（RuoYi Vue3 + Spring Boot）让前端专注交互、后端专注 API，职责清晰。若依前端已内置完整的 RBAC 管理界面，无需从零搭建。

### 3.6 不选 SQLite 作为生产数据库

- 日均 3500 笔订单 + 多客服并发写入，SQLite 的串行写锁会导致频繁阻塞。继续用 SQLite 需要在应用层自行实现写队列、重试机制、定时归档，复杂度反而高于直接使用 PostgreSQL。
- 7×24 运行下，SQLite 单文件损坏风险高，且无热备恢复手段。

### 3.7 不选 Django 替代 RuoYi

- Django 的 Admin、ORM、认证体系对 SaaS 有吸引力，但应用开发组是 Java 背景，切换 Django 等于全员重新学习。
- 已确定前后端分离，Django 最大的模板/Admin 优势被削弱。RuoYi 已提供同等且更成熟的后台管理基础设施。

## 四、部署与硬件建议

### 4.1 内网开发/测试环境

用于客服模块的开发、联调和内部验证。建议复用现有内网服务器或一台闲置 PC：

- RuoYi Java（Spring Boot Jar）
- RuoYi Vue3 前端（Nginx 或直接用 Vite DevServer）
- PostgreSQL（测试数据，可与内网 MySQL 并存）
- Redis（缓存与轻量队列）
- Python AI 服务（FastAPI + Uvicorn）

**关键注意**：内网环境**不承载真实买家流量**，仅用于开发测试。因内网无 UPS，断电会导致服务中断，无法做到真正的 7×24。

### 4.2 公网生产环境

客服模块验证成熟后，部署至公网云服务器，开始接收真实 eBay 消息。

建议以 **1 台低配比 Linux 云服务器**（如 2核 4G 起步，CentOS/Ubuntu）作为公网统一服务节点，以 Docker Compose 运行：

- Nginx（统一入口，含 RuoYi Vue3 前端静态资源托管）
- RuoYi Java（客服业务后端）
- Python AI 服务（机器人/知识库）
- PostgreSQL
- Redis
- 存量 FlaskAPI 服务（保留，与客服模块共存）

若追求高可用，可再加 1 台同规格服务器做 Nginx + Java 双节点，前置阿里云/腾讯云 CLB（负载均衡）即可，无需自建复杂集群。

### 4.3 关键配置提示

1. **内网与公网数据隔离**：内网测试的 PostgreSQL 数据不要直接同步到公网生产环境。公网发布时应重新初始化数据库，或仅迁移必要的配置数据（如机器人策略模板）。
2. **Java 内存分配**：2核 4G 服务器建议给 RuoYi Java 服务分配 `-Xms512m -Xmx1024m`，避免内存不足导致 OOM。
3. **前端部署**：RuoYi Vue3 项目 `npm run build` 后生成纯静态文件，直接放在 Nginx 的 `html/` 目录下；Nginx 对前端路由配置 `try_files` 回退到 `index.html`，支持 Vue Router 的 History 模式。
4. **WebSocket 代理**：Nginx 需开启 `proxy_http_version 1.1` 与 `proxy_set_header Upgrade $http_upgrade`，确保客服后台的实时消息通道正常穿透。
5. **Docker Compose 分服务管理**：建议将 Nginx、Java、Python AI、PostgreSQL、Redis 分为独立 Service，任一容器重启不影响其他服务。
6. **Nginx 配置拆分**：每个域名/模块使用独立的 `conf.d/` 配置文件。存量 Python API 与新增 Java 客服使用不同配置文件，方便独立维护。
7. **数据持久化**：PostgreSQL 和 Redis 的数据目录必须挂载到宿主机 Volume，避免容器重启后数据丢失。
8. **Java-Python 通信**：AI 服务建议独立暴露端口（如 `:8001`），Java 后端通过容器内网地址调用（`http://ai-service:8001`），避免走公网。

## 五、实施路线建议（渐进式）

| 阶段 | 目标 | 环境 | 周期建议 |
|------|------|------|----------|
| **Phase 1：搭建基座** | 部署 Nginx + Redis + PostgreSQL；部署 RuoYi Java 基础服务（用户/角色/菜单权限已可用）；将现有公网 FlaskAPI 接入 Nginx 反向代理，验证稳定运行。 | 公网 | 2-3 天 |
| **Phase 2：客服内网开发** | 在内网环境部署 RuoYi Java + Vue3；开发客服业务功能（消息收发、人工回复、店铺权限、机器人配置、知识库管理）；Python AI 服务同步开发机器人策略与知识库检索。 | 内网 | 4-6 周 |
| **Phase 3：客服内网验证** | 在内网模拟真实场景进行端到端测试；小范围内部试用；修复问题、优化性能，直至系统稳定。 | 内网 | 1-2 周 |
| **Phase 4：客服公网发布** | 将验证成熟的客服模块部署至公网云服务器；Nginx 新增客服域名路由；正式接收 eBay 真实买家消息，开始 7×24 在线服务。 | 公网 | 2-3 天 |
| **Phase 5：可观测性补齐** | 引入 Prometheus + Grafana 监控 Java API 与 Python AI 服务；配置企业微信告警；数据定时备份到云存储。 | 公网 | 1-2 周 |

## 六、SaaS 化演进预留

若未来计划将 eBay 智能客服等能力开放为 SaaS 对外收费，当前架构已预留扩展空间，核心只需在应用层补齐以下能力：

### 6.1 多租户隔离

- **方案**：在现有 PostgreSQL 单库中增加 `tenant_id` 字段，通过 RuoYi 数据权限机制自动拦截并追加租户过滤条件。这是成本最低的多租户方案，客户量达百级后再考虑分表或 Schema 隔离。
- **敏感数据**：eBay API Token、店铺配置等必须 AES-256 加密存储，防止租户间数据泄露。

### 6.2 订阅与配额控制

- 限制每个商家的客服坐席数、消息条数、存储时长。初期可用 Redis 计数器实现，避免引入复杂计费系统。

### 6.3 横向扩展路径

| 阶段 | 客户规模 | 关键动作 |
|------|----------|----------|
| **内部使用期** | 1 家公司 | 当前架构完全够用。 |
| **MVP 验证期** | 1-5 家（免费内测） | 应用层加 `tenant_id`；API 增加限流（Rate Limiting）。 |
| **早期收费期** | 5-50 家 | 补齐监控告警；数据定时备份到云存储。 |
| **增长期** | 50-500 家 | PostgreSQL 读写分离；Redis 主从；Java 多节点 + Nginx 负载均衡；按需引入 RabbitMQ。 |

## 七、总结

对于「深圳跨境电商行业、应用开发组以 Java 为主、成本敏感、7×24 在线」的公网服务场景，**保留现有 Python API，客服模块独立采用 RuoYi（Spring Boot）+ Vue3 开发，先内网验证再公网发布** 是当前最务实的路径。

- **存量保留**：现有公网 FlaskAPI（Token 刷新、独立站转发、企业微信代理）完全保留，由 Nginx 统一代理，与 Java 客服模块长期共存。
- **客服独立**：客服模块作为独立业务系统，由应用开发组用 Java 开发，充分利用 RuoYi 开箱即用的 RBAC 基础设施。
- **内网先行**：先在内网完成开发和验证，降低公网直接上线的风险。内网无 UPS，不承载真实 7×24 流量。
- **招聘匹配**：Java 后端在深圳给定薪酬区间下是唯一可预期招到的岗位，应用开发组现有 Java 经验可无缝复用。
- **AI 解耦**：机器人与知识库独立为 Python 服务，既顺应 AI 领域 Python 主流生态，又通过 API 与 Java 后端解耦。
- **扩展预留**：Spring Boot 单体架构在 50 家租户以内完全够用，可边盈利边迭代，无需过早引入 Spring Cloud 或 K8s。
