## 🔴 高危（必须修复）

### 1. 生产密钥硬编码提交到代码仓库
**位置**:
- `ruoyi-admin/src/main/resources/application-dev.yml`（新增 `ruoyi-wecom.qywx` 段）
- `ruoyi-admin/src/main/resources/application-prod.yml`（新增 `ruoyi-wecom.qywx` 段）

**问题**: `bot-id`、`secret`、`corp-id`、`app-secret` 等企微生产/测试凭证以明文硬编码写入 YAML 并提交到 Git。prod 中 `bot-id` 与 `secret` 与 dev 完全相同，存在泄露后被直接利用的风险。

**影响**: 密钥一旦泄露，攻击者可伪装成企微 AI 机器人接收/发送消息，或调用企微开放 API 拉取通讯录。

**修复方案**:
- 所有敏感配置统一改为环境变量注入，例如 `${QYWx_BOT_SECRET:}`、`${QYWx_CORP_ID:}`。
- 已提交到 Git 的历史明文密钥视为已泄露，**必须立即在企微后台重置**。
- 本地 dev 如需固定值，可放在 `.env` 或 `application-local.yml`（已加入 `.gitignore`）中，禁止提交到仓库。

### 2. `/qywx/**` 全局匿名访问 + 管理端点无鉴权
**位置**:
- `ruoyi-framework/src/main/java/com/ruoyi/framework/config/SecurityConfig.java:112`
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/controller/QywxController.java`

**问题**:
1. `SecurityConfig` 新增 `.antMatchers("/qywx/**").permitAll()`，导致所有 `/qywx/` 路径对匿名用户开放。
2. `QywxController` 整体标注 `@Anonymous`，且各方法无 `@PreAuthorize` 等 RBAC 控制。
3. 该 Controller 暴露的是**运维管理接口**（列出机器人、手动连接/断开 WebSocket），而非企微回调入口。企微消息回调走的是 WebSocket 长连接，并不经过 Spring Security 的 HTTP 过滤器链。

**影响**: 任何能访问该服务的人都可以调用 `POST /qywx/ai-bot/{code}/disconnect` 断开生产机器人连接，或 `POST /qywx/ai-bot/{code}/connect` 建立非预期连接，造成服务中断或消息被异常消费。

**修复方案**:
- 删除 `SecurityConfig` 中的 `.antMatchers("/qywx/**").permitAll()`。
- `QywxController` 去掉类级别 `@Anonymous`。
- 各端点按需添加鉴权注解，例如：
  ```java
  @PreAuthorize("@ss.hasPermi('wecom:bot:manage')")
  ```
- 若确实存在需要匿名访问的端点（如健康检查、企微 URL 验证回调），应缩小到具体路径，而非整个 `/qywx/**`。

## 🟠 严重（推荐修复）

### 3. 配置与代码不一致：`list-url`、`list-max-jobs` 被 YAML 配置但 Java 类未定义
**位置**:
- `ruoyi-admin/src/main/resources/application-*.yml`
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/config/QywxProperties.java`

**问题**: YAML 中配置了 `job-api.list-url` 与 `job-api.list-max-jobs`，但 `QywxProperties.JobApi` 类中只有 `runUrl`、`token`、`tokenFile` 等字段，**没有 `listUrl` 和 `listMaxJobs` 的 setter**。Spring Boot 在绑定时会静默忽略这些未匹配的属性。代码中也未调用 astpy `/jobs/list` 接口。

**影响**: 运维人员看到配置项以为功能已可用，实际无效；属于功能不完整或配置残留。

**修复方案**:
- 若「任务列表」意图从 astpy 拉取：补充 `listUrl` / `listMaxJobs` 字段并实现调用逻辑。
- 若不需要：从 YAML 中删除相关配置，避免误导。

---

### 4. 每次请求都新建 `RestTemplate` 且同步读文件取 Token
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/service/impl/QywxJobApiServiceImpl.java`

**问题**:
1. `post()` 方法内部每次都 `new RestTemplate(factory)`，未复用连接池。
2. `resolveToken()` 在 `config.getToken()` 为空时，每次都调用 `readTokenFromFile()` 从磁盘读取 env 文件。

**影响**: 高频命令触发时频繁创建 HTTP 客户端和磁盘 IO，增加延迟与资源消耗；极端情况下可能触发文件句柄或线程数问题。

**修复方案**:
- 将 `RestTemplate` 提升为类成员或 Bean，通过 `RestTemplateBuilder` 统一配置。
- Token 读取增加内存缓存（如 Guava Cache / Caffeine），按 30s~60s TTL 刷新，避免每次请求读盘。

---

### 5. `QywxOpenUserIdServiceImpl` 使用 `synchronized (this)` 刷新 access_token
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/service/impl/QywxOpenUserIdServiceImpl.java`

**问题**: 在 Spring 单例上 `synchronized (this)`，所有请求 `resolveContactUserId` 的线程在 token 过期时会被串行阻塞。

**影响**: 并发消息处理时产生不必要的排队；虽然单次 token 请求很快，但在高并发场景下会放大延迟。

**修复方案**:
- 使用更细粒度的锁（如 `ReentrantLock` 或专门的 token refresh 对象锁），或采用无锁的 `AtomicReference<TokenHolder>` + CAS 重试模式。

---

### 6. 管理端点缺少权限校验，即使修复 `permitAll` 后仍需 RBAC
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/controller/QywxController.java`

**问题**: 若保留为管理员接口，当前没有任何方法级权限注解。即使去掉 `permitAll`，任何已登录用户都能连接/断开机器人。

**影响**: 内部非管理员用户可操作生产机器人。

**修复方案**:
- 为 `connect` / `disconnect` / `listAiBots` 添加 `@PreAuthorize("@ss.hasPermi('wecom:bot:manage')")` 或类似角色控制。

## 🟡 一般（建议修复）

### 7. `SecurityConfig` 中 `/qywx/**` 路径放通过于宽泛
**位置**:
- `ruoyi-framework/src/main/java/com/ruoyi/framework/config/SecurityConfig.java`

**问题**: 使用前缀匹配会覆盖未来所有新增在 `/qywx/` 下的端点，容易导致后续开发误以为默认安全。

**修复方案**:
- 删除全局前缀放通，改为在具体 Controller/方法上使用 `@Anonymous` 或 `@PreAuthorize` 精确控制。

---

### 8. `QywxAiBotClient.sendMarkdownMessage` 存在竞态条件
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/client/QywxAiBotClient.java:614`

**问题**: `sendMarkdownMessage` 非 `synchronized`，先检查 `isConnected()` 再调用 `connect()` 非原子操作；多线程同时调用时可能触发多次连接或 `NullPointerException`。

**修复方案**:
- 将连接检查和发送逻辑合并到同一个同步块内，或复用 `sendWithAck` 的锁机制。

---

### 9. `parseCommand` 吞掉所有异常
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/service/impl/QywxCommandServiceImpl.java`

**问题**: `parseCommand` 的 `catch (Exception e)` 直接返回 `UNKNOWN_COMMAND_MESSAGE`，丢失了真实异常栈。

**修复方案**:
- 区分「已知业务异常」与「系统异常」：系统异常应抛出或记录 ERROR 级别日志，而非当作普通未知命令处理。

---

### 10. 请求 ID 生成器存在低概率冲突
**位置**:
- `ruoyi-wecom/src/main/java/com/ruoyi/wecom/util/QywxJobRequestIdGenerator.java`

**问题**: 格式 `yyyyMMddHHmmss + 4位随机数`，同一秒内 10000 个并发时冲突概率不可忽略。

**修复方案**:
- 追加进程/线程标识，或使用 `UUID.randomUUID().toString().replace("-", "").substring(0, 16)` 等高熵方案。

---

### 11. 新增模块无任何自动化测试
**位置**: `ruoyi-wecom` 全模块

**问题**: 核心流程（命令解析、权限校验、Job API 调用、WebSocket 帧处理）均未覆盖单元测试或集成测试。

**修复方案**:
- 至少补充：
  - `QywxCommandServiceImpl.parseCommand` 的边界/异常用例。
  - `QywxUserPermissionServiceImpl` 的权限判定矩阵。
  - `QywxJobApiServiceImpl` 对 astpy 接口的 Mock 调用与超时/重试场景。
