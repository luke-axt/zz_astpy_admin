## 🔴 高危（必须修复）

### 1. KeyError 风险 — 橙联接口返回额外产品码导致崩溃

**位置：** `rpa/lastmile/LastMileByLingxing.py` 第 870 行附近（PR diff）

**问题代码：**
```python
plan_price_dict[(warehousecode, prod, ship_method_typeid_map[(warehousecode,prod)])] = float(price_total_usd)
```

**问题：** `ship_method_typeid_map` 是从 `oc_usable_planlist` 构建的映射 `(warehousecode, plancode) -> typeid`。如果橙联批量测算接口返回了不在原始请求列表中的 `productOuterCode`（即 `prod`），则 `(warehousecode, prod)` 不在 `ship_method_typeid_map` 中，直接抛 `KeyError`。

**影响：** 单个渠道的异常数据会导致整个订单处理流程崩溃，中断所有尾程渠道的测算。

**修复方案：**
```python
key = (warehousecode, prod)
if key in ship_method_typeid_map:
    plan_price_dict[(warehousecode, prod, ship_method_typeid_map[key])] = float(price_total_usd)
else:
    self.logger.warning(f"橙联返回未请求的产品码: {prod}, 仓库: {warehousecode}")
```

### 2. 接口文档与实际代码不一致 — `get_lm_pricedetail` 仍声明支持 `oc`

**位置：** `CoreBussiness/LmCore.py` 第 74 行（docstring）

**问题代码：**
```python
info_dict['lm_comp']   jf/wit/lc/gc/oc
```

**问题：** 虽然 docstring 明确列出 `oc` 为支持的参数值，但实际代码中 `lm_comp == 'oc'` 的分支已被整段注释掉（第 288 行起）。如果调用方仍按文档传入 `oc`，会走到最后的 `raise RuntimeError`，导致运行时异常。

**影响：** 其他模块开发者可能仍按旧文档调用，导致生产环境运行时错误。

**修复方案：** 更新 `get_lm_pricedetail` 的 docstring，将 `oc` 从 `lm_comp` 支持列表中移除，并补充说明橙联请使用 `get_lm_pricedetail_by_oc`。

### 4. 错误码列表中包含空字符串导致误判

**位置：** `CoreBussiness/LmCore.py` 第 438 行

**问题代码：**
```python
if errormsg in ('分单逻辑计算失败','系统未匹配到合适的包材，请联系客服','无可用物流服务。','','匹配包材类型异常',...):
```

**问题：** 错误码元组中包含了空字符串 `''`。如果橙联接口返回的 `errorMessage` 为空字符串，会被误判为已知业务错误（返回 `0020001`），而不是未知错误（返回 `0010001`）。

**影响：** 可能导致业务错误被错误归类，影响上游的错误处理和告警策略。

**修复方案：** 从错误码列表中移除空字符串 `''`。

### 5. `plan_list` 为空时仍返回成功状态码

**位置：** `CoreBussiness/LmCore.py` 第 448 行

**问题代码：**
```python
for fee_item in fee_data_list:
    if fee_item['success']:
        ...
        plan_list.append({...})
return '0', '调用成功', plan_list
```

**问题：** 如果 `fee_data_list` 中所有 `fee_item['success']` 均为 `False`，`plan_list` 为空列表 `[]`，但函数仍然返回 `'0'`（成功）。

**影响：** 调用方无法区分"接口调用成功且有可用方案"与"接口调用成功但无可用方案"，可能导致下游基于空列表做错误决策。

**修复方案：** 在返回前判断 `plan_list` 是否为空，若为空返回特定业务错误码（如 `0020001`）并附带说明。

## 🟡 一般（建议修复）

### 7. 多错误场景下只处理第一个错误

**位置：** `CoreBussiness/LmCore.py` 第 436~441 行

**问题代码：**
```python
for error in fee_result_info.get_data()['errors']:
    errormsg = error['errorMessage']
    if errormsg in (...):
        return '0020001', errormsg, plan_list
    else:
        return '0010001', '测算接口出现未知错误。' + ..., plan_list
```

**问题：** 循环在第一个 error 处就直接 `return`，不会检查后续 errors。如果存在混合错误（第一个已知、第二个未知，或反之），处理结果取决于错误顺序，不稳定。

**影响：** 错误处理逻辑在边界情况下表现不一致。

**修复方案：** 先遍历全部错误进行分类，若全部已知则返回 `0020001`（可拼接所有已知错误信息），若存在任一未知错误则返回 `0010001`。
