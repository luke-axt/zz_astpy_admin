## 🟠 严重（推荐修复）

### 2. `AstdcService.send_qw_msg` 硬编码敏感信息，且作为公共方法扩大影响范围  ok
- **位置：** `CoreBussiness/AstdcService.py:219-234`
- **问题：** 本次 PR **新增**的公共方法 `send_qw_msg` 直接硬编码 `agentid: 1000022` 和 `qywx_add_token="BFj9jQo4t4OV1Ni8bNWXxRNLfDi"`。相比旧代码仅在单个 Job 内硬编码，现在提升至 `AstdcAPI` 公共 Service 层，**任何调用方都会使用同一套固定凭证**。
- **影响：**
  1. 敏感信息硬编码在公共层，泄露风险更大。
  2. 其他业务模块若调用此方法，只能使用这个固定的 `agentid` 和 `token`，无法按业务场景灵活配置。
  3. `Token` 过期或变更时需修改公共 Service 代码，影响面扩大。
- **修复方案：** 将 `agentid` 和 `qywx_add_token` 抽取到 `adminconfig` 配置模块或环境变量中，通过配置读取：
  ```python
  def send_qw_msg(self, touser, msg, msgtype='text', agentid=None, qywx_add_token=None):
      agentid = agentid or adminconfig.QYWX_AGENT_ID
      qywx_add_token = qywx_add_token or adminconfig.QYWX_ADD_TOKEN
      ...
  ```

### 3. `AdlEbaySuspectOrderMonitorJob.send_qywx_message` 异常处理使用裸 `except` 且异常时不重试 ok
- **位置：** `dwd/adl/ebay/AdlEbaySuspectOrderMonitorJob.py:131-148`
- **问题：**
  1. 使用裸 `except:` 捕获所有异常（包括 `KeyboardInterrupt`、`SystemExit` 等）。
  2. `except` 块内没有 `time.sleep(retry_interval)` 和 `continue`，一旦发生异常（如网络超时、DNS 解析失败），直接退出 `for` 循环，**重试机制完全失效**。
- **影响：** 网络临时抖动时消息发送直接失败，不会进行重试。
- **修复方案：**
  ```python
  except Exception as e:
      err_msg = f"未知报错：{traceback.format_exc()}"
      time.sleep(retry_interval)
      continue
  ```

### 4. `AstdcService.send_qw_msg` 的 `msgtype` 参数与消息体结构不匹配 ok
- **位置：** `CoreBussiness/AstdcService.py:219-234`
- **问题：** 方法签名支持 `msgtype='text'`，但 `json_param` 中固定使用 `"text": {"content": msg}`。若调用方传入 `msgtype='markdown'`，企微 API 实际仍按 `text` 类型发送，参数与行为不一致。
- **影响：** 接口存在设计缺陷，后续若扩展为 `markdown` 或其他消息类型时会产生 Bug。
- **修复方案：** 根据 `msgtype` 动态构建消息体：
  ```python
  json_param = {
      "touser": touser,
      "msgtype": msgtype,
      "agentid": agentid,
      "safe": 0,
      "enable_id_trans": 0,
      "enable_duplicate_check": 0
  }
  if msgtype == 'text':
      json_param["text"] = {"content": msg}
  elif msgtype == 'markdown':
      json_param["markdown"] = {"content": msg}
  ```

## 🟡 一般（建议修复）

### 5. 无新增订单时仍发送企微通知，产生噪音
- **位置：** `dwd/adl/ebay/AdlEbaySuspectOrderMonitorJob.py:102-104`
- **问题：** `ebay_susporder_df.empty` 时仍调用 `self.send_qywx_message("本次无新增可疑订单")`。
- **影响：** 每次 Job 执行都推送无意义通知，消耗企微 API 调用配额，降低重要通知的关注意愿。
- **修复方案：** 删除该通知调用，仅保留日志记录：`self.logger.info("本次无新增可疑订单")`。

### 6. `select_new_order` 使用 `SELECT *`
- **位置：** `dwd/adl/ebay/AdlEbaySuspectOrderMonitorJob.py:122-125`
- **问题：** 查询语句使用 `SELECT *`，未明确指定所需列。
- **影响：** 表结构变更时可能导致下游处理异常；不必要的列增加网络传输。
- **修复方案：** 改为明确列名：`SELECT ordercode, receive_addr, buyer_email, buyerid, paidtime, skuinfo, createtime FROM ...`。

### 7. `select_new_order` 存在性能风险
- **位置：** `dwd/adl/ebay/AdlEbaySuspectOrderMonitorJob.py:122-125`
- **问题：** `WHERE createtime > NOW() - INTERVAL 20 MINUTE` 依赖 `createtime` 列索引。若 `astdc.dc_yy_hg_susporder` 表数据量大且无索引，将触发全表扫描。
- **影响：** 查询性能差，拖慢 Job 执行。
- **修复方案：** 确认 `createtime` 是否已建立索引；如无，建议添加 `INDEX idx_createtime (createtime)`。
