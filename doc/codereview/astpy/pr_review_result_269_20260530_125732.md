## 🔴 高危（必须修复）

### 1. `refresh_bal_data` 方法成功时无返回值，导致 `action` 崩溃 -- ok
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:67-76`
- **问题：** 方法仅在 `res.is_fail()` 时 `return res`，成功分支没有显式返回，隐式返回 `None`。
- **影响：** `action` 方法中 `res = self.refresh_bal_data(count_day)` 后执行 `if res.is_fail():` 会抛出 `AttributeError: 'NoneType' object has no attribute 'is_fail'`，导致整个 Job 直接崩溃，无法继续执行后续逻辑。
- **修复方案：**
  ```python
  def refresh_bal_data(self, count_day):
      curdate = DatePack.parseDatetime2Str(DatePack.getCurtime(), DatePack.YYYYMMDD)
      paramJson = f"""[{{"format":"{{dt}}","value":"{curdate}"}},{{"format":"{{count_day}}","value":"{count_day}"}}]"""
      res = DwService('bal').run_job('bal_ebay_susporder_info', paramJson)
      if res.is_fail():
          return res
      return ResultObj.success()  # <-- 添加此行
  ```

### 2. `run_adl_push_data` 方法成功时无返回值，导致 `action` 崩溃  ok
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:78-87`

## 🟠 严重（推荐修复）

### 3. `send_qywx_message` 异常处理使用裸 `except` 且异常时不重试  忽略
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:131-155`
- **问题：**
  1. 使用裸 `except:` 捕获所有异常（包括 `KeyboardInterrupt`、`SystemExit` 等系统异常）。
  2. `except` 块内没有 `time.sleep(retry_interval)` 和 `continue`，一旦发生异常（如网络超时、DNS 解析失败），直接退出 `for` 循环，**重试机制完全失效**。
- **影响：** 网络临时抖动时消息发送直接失败，不会进行重试；裸 `except` 可能掩盖系统退出信号。
- **修复方案：**
  ```python
  def send_qywx_message(self, msg):
      max_retry = 3
      retry_interval = 1
      err_msg = ""
      for i in range(max_retry):
          try:
              action = "https://qyapi.weixin.qq.com/cgi-bin/message/send"
              json_param = {
                  "touser": self.touser,
                  "msgtype": "text",
                  "agentid": 1000022,
                  "text": {"content": msg},
                  "safe": 0,
                  "enable_id_trans": 0,
                  "enable_duplicate_check": 0
              }
              res = self.astdc_api.call_qywx_serv_api(
                  action=action,
                  qywx_add_token="BFj9jQo4t4OV1Ni8bNWXxRNLfDi",
                  json_param=json_param
              )
              if res.is_fail():
                  err_msg = f"企微通知接口调用失败：{res}"
                  time.sleep(retry_interval)
                  continue

              datajson = json.loads(res.get_data())
              if datajson["errcode"] != 0:
                  err_msg = f"企微接口发送信息失败：{res}"
                  time.sleep(retry_interval)
                  continue

              return ResultObj.success()
          except Exception as e:  # <-- 改为 Exception
              err_msg = f"未知报错：{traceback.format_exc()}"
              time.sleep(retry_interval)  # <-- 添加睡眠
              continue  # <-- 添加 continue，确保重试

      return ResultObj.error(ResultObj.FATAL_ERROR, err_msg)
  ```

### 4. 企业微信敏感信息（Token 和 AgentID）硬编码  ok
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:137-138, 147`
- **问题：** `qywx_add_token="BFj9jQo4t4OV1Ni8bNWXxRNLfDi"` 和 `agentid: 1000022` 直接硬编码在源码中。旧代码同样存在此问题，重构时未做安全整改。
- **影响：**
  1. 敏感信息泄露风险（代码仓库、日志中可能暴露）。
  2. Token 过期或变更时必须修改源码并重新部署，灵活性差。
- **修复方案：** 将 `qywx_add_token` 和 `agentid` 抽取到配置中心、环境变量或 `adminconfig` 配置模块中，通过配置读取。

## 🟡 一般（建议修复）

### 5. 无新增订单时仍发送企微通知，产生噪音  忽略
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:102-104`
- **问题：** `ebay_susporder_df.empty` 时仍调用 `self.send_qywx_message("本次无新增可疑订单")`，每次 Job 执行都会推送一条无意义的通知。
- **影响：** 企微通知通道产生大量噪音，降低真正可疑订单通知的关注意愿，且消耗企微 API 调用配额。
- **修复方案：** 删除该通知调用，仅记录日志即可：`self.logger.info("本次无新增可疑订单")`。

### 6. `select_new_order` 使用 `SELECT *`   忽略
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:122-125`
- **问题：** 查询语句使用 `SELECT *`，未明确指定所需列。
- **影响：** 表结构变更时可能导致下游处理异常；不必要的列会增加网络传输开销。
- **修复方案：** 改为明确列名，如 `SELECT ordercode, receive_addr, buyer_email, buyerid, paidtime, skuinfo, createtime FROM ...`。

### 7. `select_new_order` 存在性能风险   忽略
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:122-125`
- **问题：** `WHERE createtime > NOW() - INTERVAL 20 MINUTE` 依赖 `createtime` 列的索引。如果 `astdc.dc_yy_hg_susporder` 表数据量大且 `createtime` 无索引，每次查询将触发全表扫描。
- **影响：** 查询性能差，可能拖慢整个 Job 执行时间，高并发时加重数据库负载。
- **修复方案：** 确认 `createtime` 列是否已建立索引；如无索引，建议添加 `INDEX idx_createtime (createtime)`。

### 8. `ebay_suspect_order_send` 中对 DataFrame 使用 `iterrows()` 遍历  忽略
- **位置：** `dwd/adl/ebay/EbaySuspectOrderMonitorJob.py:99-115`
- **问题：** `iterrows()` 遍历 DataFrame 效率低，且每行调用多次 `.get()` 方法，存在额外开销。
- **影响：** 可疑订单量大时，消息拼接耗时增加。
- **修复方案：** 使用 `apply` 或 `to_dict('records')` 方式优化，或考虑使用列表推导式拼接字符串。

## 审查结论

### ❌ 不允许合并

**理由：** 本次 PR 存在 **2 个高危问题**，会导致 `EbaySuspectOrderMonitorJob` 在生产环境直接崩溃：
1. `refresh_bal_data` 方法成功时返回 `None`，后续 `res.is_fail()` 抛出 `AttributeError`。
2. `run_adl_push_data` 方法成功时返回 `None`，同样会导致崩溃。

这两个问题属于 **功能性缺陷**，一旦合并上线，Job 执行必然失败。

**必须先修复以下问题后再合并：**
- [ ] 修复 `refresh_bal_data` 缺少返回值的问题
- [ ] 修复 `run_adl_push_data` 缺少返回值的问题

- [ ] 建议整改企业微信 Token 和 AgentID 硬编码的安全隐患
- [ ] 建议确认 SQL 条件删除的业务影响
