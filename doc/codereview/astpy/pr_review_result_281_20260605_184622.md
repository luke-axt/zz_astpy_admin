## 审查发现

### 🔴 高危（必须修复）

#### 1. `OcOrder.py:118-120` — 移除失败订单降级保护，可能导致任务被永久阻塞

**位置：** `ETL/ods/OrangeConnexOds/OcOrder.py`，`update_order_without_trac_data` 方法

**问题：** 旧逻辑在 `jsapi_outbound_order_postcode` 3 次重试仍然失败时，会调用 `self.update_trac_order_not_trac_data(order_number)` 将该订单的 `trac_update_time` 更新为 `now()`，随后 `continue` 跳过该订单。新逻辑直接 `return ResultObj.error(...)`，不再执行此降级保护。

**影响：**
- `select_oc_outbound_order_trac_detail_without_trac_data` 按 `trac_update_time asc` 排序，且 `trac_update_time` 初始值为 `'2000-01-01 00:00:00'`。
- 如果某个订单因持续性原因（如订单号在橙联系统已删除、格式异常）导致 API 失败，其 `trac_update_time` 永远不会被更新，该订单将永远排在待查询队列最前端。
- 每次任务执行到此订单都会触发 `FATAL_ERROR` 并 return，后续所有订单永远无法被处理，任务陷入永久性失败循环。

**修复方案：**
在 `return ResultObj.error(...)` **之前**，保留对当前失败订单的 `self.update_trac_order_not_trac_data(order_number)` 调用，避免该订单反复阻塞后续任务：

```python
if postcode_res.is_fail():
    self.update_trac_order_not_trac_data(order_number)
    return ResultObj.error(ResultObj.FATAL_ERROR, f"查询轨迹信息失败，出库订单号：{order_number}，{postcode_res.get_msg()[:200]}")
```

或者，在 `action` 方法或调度层增加"失败订单跳过/告警"机制，防止单个坏数据瘫痪整条流水线。

#### 2. `OcInv.py:48` — 单区域失败导致全量数据丢弃，系统容错性严重退化

**位置：** `ETL/ods/OrangeConnexOds/OcInv.py`，`ImportOCStockInfoByJs.action` 方法

**问题：** 旧逻辑在 `jsapi_purchase_history` 3 次重试失败后，执行 `break` 跳出当前区域的 `while` 循环，继续采集下一个区域，最终将所有成功区域的数据统一 `delete` + `insert` 入库。新逻辑改为直接 `return ResultObj.error(...)`，丢弃 `insert_data_list` 中已累积的全部数据。

**影响：**
- 橙联库存采集覆盖 5 个区域（DE、UK、US-West、US-East、US-South）。如果其中一个冷门区域因临时网络抖动或对方系统维护而失败，其余 4 个区域已采集的数据也会被一并丢弃。
- 由于 `delete_stock_info()` 和批量 `insert` 位于 `return` 之后，数据库将长期保留上一次（甚至上一天）的旧数据，造成数据新鲜度严重下降。
- 相比旧逻辑的"部分可用"策略，新逻辑将系统的容错能力从"允许单点故障"降级为"零容忍任何故障"。

**修复方案：**
建议恢复旧逻辑的"跳过失败区域"策略，或增加配置开关，允许按区域容错。如果业务确实要求"全有或全无"，则应至少：
1. 在采集开始前先对 `insert_data_list` 做内存校验，确保 5 个区域全部采集成功后再执行 `delete` + `insert`（当前代码结构已满足这一点，但 `return` 时机过早，导致内存数据浪费）。
2. 在告警/监控层面明确通知"部分区域失败导致全量未更新"，避免运维人员误以为数据正常。

### 🟠 严重（推荐修复）

#### 3. `OcOrder.py` — 逐条更新已造成部分落库，"丢弃本次全部数据"与实际行为不符

**位置：** `ETL/ods/OrangeConnexOds/OcOrder.py`，`update_trac_order_zone` 与 `update_order_without_trac_data` 方法

**问题：** 两个方法均在 `for` 循环内逐条调用 `update_oc_outbound_order_trac_detail_zone` 和 `update_oc_outbound_order_postcode`，即每条更新都实时提交到数据库。当第 N 条记录触发失败并 `return` 时，第 1 至 N-1 条记录的实际更新已经落库。但返回的错误消息和日志描述为"丢弃本次全部数据"，这与实际行为不符。

**影响：**
- `action` 方法在收到失败结果后会直接返回，导致 `update_order_without_trac_data` 或后续步骤不再执行。
- 系统处于不一致状态：例如部分订单的 `zone` 已更新但 `postcode` / `delivery_time` 未更新；或者部分订单已更新，另一部分没有。
- 日志误导运维人员以为没有任何数据被修改，实际上数据库已被部分更新。

**修复方案：**
- **方案 A（推荐）：** 在 `return ResultObj.error(...)` 的日志和错误消息中，明确说明"已部分更新 N 条记录，因订单 X 失败而中止后续处理"，消除误导。
- **方案 B：** 如果业务严格要求"全有或全无"，需引入数据库事务包裹整个循环（注意长时间事务对大表更新的影响），或在内存中累积全部更新参数，待全部采集成功后再批量执行 `UPDATE`。

### 🟡 一般（建议修复）

#### 4. `OcInv.py` — 已采集数据被丢弃，造成重复采集开销

**位置：** `ETL/ods/OrangeConnexOds/OcInv.py`，`ImportOCStockInfoByJs.action` 方法

**问题：** 假设 DE 区域已成功采集 500 条记录到 `insert_data_list`，随后 UK 区域第 1 页请求失败，直接 `return`，导致 DE 的 500 条内存数据被 Python GC 回收。

**影响：**
- JS 自动化采集本身成本较高（涉及浏览器启动、登录、页面交互）。已付出的采集成本因后续区域失败而完全浪费。
- 下一次任务重试时，DE 区域需要重新采集。

**修复方案：**
如果确认维持"失败即中止"策略，建议在日志中补充记录"已采集但丢弃的数据量"，便于后续分析采集成本与失败率：

```python
return ResultObj.error(ResultObj.FATAL_ERROR, f"采集橙联【{region}】区域库存失败，已采集但丢弃 {len(insert_data_list)} 条数据：{res.get_msg()[:200]}")
```

### 🟢 优化（可选）

#### 5. `OcInv.py:48` / `OcOrder.py:86` — `get_msg()[:200]` 存在潜在 `TypeError`

**位置：** 多处涉及 `res.get_msg()[:200]` 或 `postcode_res.get_msg()[:200]` 的代码行

**问题：** 如果 `ResultObj.get_msg()` 在某些异常路径下返回 `None`，直接对 `None` 进行切片操作将抛出 `TypeError: 'NoneType' object is not subscriptable`。

**影响：**
- 虽然当前代码中 `jsapi_*` 方法在失败时似乎总会返回字符串消息，但防御性编程不足。如果底层 API 或 `ResultObj` 未来发生变化，可能引入运行时异常。

**修复方案：**
顺手改为空值安全写法：

```python
msg = res.get_msg() or ""
# 或使用 str()
return ResultObj.error(..., f"...{str(res.get_msg())[:200]}")
```

#### 6. `OcOrder.py:97` / `OcOrder.py:149` — 日志表述建议调整

**位置：** `update_trac_order_zone` 与 `update_order_without_trac_data` 方法的末尾日志

**问题：** 新逻辑下日志改为 `"成功 {total}/{total}"` 和 `"成功 {total}条"`。虽然如果执行到该行说明循环已完整跑完（全部成功），但若未来代码被修改或异常路径被调整，该日志不再包含失败计数，信息量降低。

**修复方案：**
保持 `"成功 X / 总计 Y"` 的日志格式，将失败数显示为 0，既与新逻辑匹配，也保留了信息量：

```python
self.logger.info(f"...订单分区更新结束，成功 {total}/{total}，失败 0")
```

## 审查结论

### ❌ 不允许合并

**原因：**
1. **高危问题 #1** 直接威胁系统可用性：单个有问题的订单即可导致整个 ETL 任务循环失败，后续所有订单永远无法被处理。这是一个典型的"坏数据阻塞好数据"的可用性灾难。
2. **高危问题 #2** 大幅降低了 `OcInv.py` 的容错能力：橙联 5 个区域中任何一个偶发故障，都会导致全量库存数据无法更新，运维压力剧增。
3. **严重问题 #3** 造成数据状态不一致与日志误导：代码声称"丢弃全部"，实际已经部分更新，给故障排查带来极大困扰。

**建议修复后再审：**
- 在 `OcOrder.py:update_order_without_trac_data` 的 `return` 前恢复 `update_trac_order_not_trac_data(order_number)` 调用，或在调度层实现"坏订单跳过/告警"机制。
- 重新审视 `OcInv.py` 的"单区域失败即全量丢弃"策略，建议恢复区域级容错，或提供配置化开关。
- 统一修正 `OcOrder.py` 的日志表述，明确告知"已部分更新"的事实，避免运维误导。
