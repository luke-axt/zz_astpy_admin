## 🔴 高危（必须修复）

### 1. NameError：异常处理块引用未定义变量

- **位置**: `CoreBussiness/GuCangService.py:446`
- **问题**: 登录重试循环中，`escaped_html` 在 `try` 块内部定义（第 443 行），但 `except` 块（第 446 行）直接引用它。如果异常发生在 `find_element`、`send_keys`、`execute_script` 等阶段（即 `save_screenshot` 之前），`escaped_html` 尚未赋值，会导致 `NameError`。
- **影响**: 登录失败时的错误处理逻辑本身崩溃，原始异常被 `NameError` 掩盖，运维人员无法从日志中获取有效排错信息。
- **修复方案**:
  ```python
  escaped_html = ""
  for i in range(3):
      try:
          # ... 登录操作 ...
          browser.save_screenshot(self.save_img)
          page_html = browser.page_source
          escaped_html = html.escape(page_html[:5000])  # 同时限制长度
          WebDriverWait(browser, 20).until(...)
          return ResultObj.success(browser)
      except Exception:
          msg = f'登录谷仓系统异常，错误信息：{traceback.format_exc()}\n截图：{self.save_img}\nhtml：{escaped_html}'
          self.logger.error(msg)
  ```

### 2. 方法重命名与动态调用不兼容

- **位置**: `task/modules/TaskLX.py:313`
- **问题**: `lx_bundled_sku_match` 被重命名为 `lx_multi_platform_sku_pair`。本项目任务调度通过 `JobCore.dynamic_import_class_method_call` 使用 `getattr(instance, method_name)` 动态调用，而 `method_name` 来自数据库/配置中的 `component` 字段（格式：`模块名:类名.方法名`）。代码仓库中未同步更新任何配置引用。
- **影响**: 如果数据库配置中仍为旧方法名 `lx_bundled_sku_match`，定时任务执行时将抛出 `AttributeError`，导致业务任务全部失败。
- **修复方案**:
  - 方案 A（推荐）：在数据库/配置同步更新前，保留旧方法名作为别名过渡：
    ```python
    def lx_bundled_sku_match(self, **kwargs):
        """兼容旧配置名"""
        return self.lx_multi_platform_sku_pair(**kwargs)
    ```
  - 方案 B：确认并同步更新 `dc_job` 或相关配置表中所有 `component` 字段指向新方法名。

### 3. 空值调用 `.lower()` 导致 AttributeError

## 🟠 严重（推荐修复）

### 4. 硬编码绝对路径且文件覆盖

- **位置**: `CoreBussiness/GuCangService.py:28`
- **问题**: `self.save_img = r'D:\data\GuCangError.png'` 为 Windows 绝对路径，不具备可移植性；且 3 次重试均写入同一文件，后一次覆盖前一次。
- **影响**: 多实例部署时路径可能不存在；无法保留每次重试的截图用于对比分析；多进程同时写同一文件存在竞争。
- **修复方案**: 使用配置中心或环境变量获取路径；文件名加入时间戳和进程标识，如 `f"{base_path}/GuCangError_{DatePack.getCurtime()}_{os.getpid()}.png"`。

### 5. 裸异常捕获 `except:`

- **位置**: `CoreBussiness/GuCangService.py`（多处，如第 421、446 行）
- **问题**: 使用裸 `except:` 捕获所有异常，包括 `KeyboardInterrupt`、`SystemExit` 等系统级异常。
- **影响**: 可能意外吞掉系统信号，导致程序无法正常终止；调试困难。
- **修复方案**: 统一改为 `except Exception:`，如需捕获特定异常（如 `WebDriverException`、`TimeoutException`）应进一步细化。

### 6. 日志中写入完整 HTML 源码

- **位置**: `CoreBussiness/GuCangService.py:446`
- **问题**: `html.escape(page_html)` 将整个浏览器页面 HTML 转义后写入日志。单页面可能数 MB，经 `html.escape` 后体积更大。
- **影响**: 日志文件急剧膨胀，磁盘 I/O 开销大；集中式日志系统可能截断或拒绝接收；排查时信息过载。
- **修复方案**: 仅保留关键片段（如 `<title>`、错误提示元素），或限制长度：`html.escape(page_html[:5000])`。

## 🟡 一般（建议修复）

### 7. 重试错误信息被覆盖

- **位置**: `CoreBussiness/GuCangService.py:446`
- **问题**: `msg` 在每次 `except` 块中被完全重新赋值，循环结束后只保留了最后一次失败的错误信息。
- **影响**: 无法了解前几次重试的失败原因，丢失排线索。
- **修复方案**: 累积记录每次失败，或在每次 `except` 中独立调用 `logger.error`，最终 `ResultObj.error` 返回汇总信息。

### 8. 接口返回类型变更未确认兼容性

- **位置**: `rpa/lingxing/LxParseMskuJob.py:696`
- **问题**: `return ResultObj.success()` 改为 `return ResultObj.success(import_data_list)`，返回体从无数据变为携带列表。
- **影响**: 若上游调用方在 `is_success()` 后未预期 `get_data()` 有值，可能无异常，但行为已变；若调用方对返回值有校验，可能失败。
- **修复方案**: 全局搜索并确认 `ImportLxPlatformMapDataUnmatch.run()` 或 `LxParseMskuJob.run()` 的所有调用方兼容此变更。

### 9. URL 查询参数末尾多余 `&`

- **位置**: `rpa/lingxing/LxParseMskuJob.py:665`
- **问题**: `shop_params = "".join(f"store_id[]={sid}&" for sid in array)` 生成的字符串末尾带 `&`，与后续 `offset=...` 拼接后 URL 为 `...store_id[]=x&&offset=...`。
- **影响**: 大多数 HTTP 服务端可容错，但不符合 RFC 规范，存在潜在兼容风险。
- **修复方案**: 使用标准库拼接：`urllib.parse.urlencode([(f"store_id[]", sid) for sid in array], doseq=True)`。

### 10. 宽泛异常捕获掩盖数据格式问题

- **位置**: `rpa/lingxing/MarkLxEbayMsgAsHandledJob.py:124`
- **问题**: `except Exception:` 同时覆盖了"元素不存在"和"正则匹配失败"两类不同问题。如果 `num_btn.text` 格式不符合预期（如文本中无 `(数字)`），`re.search(...).group(1)` 的 `AttributeError` 会被当作"无待处理留言"处理。
- **影响**: 服务端数据格式变化时，异常被静默吞掉，业务逻辑以"无数据"继续执行，可能导致漏处理。
- **修复方案**: 分别捕获 `NoSuchElementException`（无按钮→num_order=0）和正则异常（格式异常→记录警告日志）。
