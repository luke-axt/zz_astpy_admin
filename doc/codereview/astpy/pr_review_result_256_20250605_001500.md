## 审查发现

### 🔴 高危（必须修复）

#### 1. `_login_with_retry` 缺少异常保护，可能导致 UnboundLocalError
- **位置**: `ETL/ods/OrangeConnexOds/OcInv.py:112-130` (`_login_with_retry`)
- **问题**: `self.service.loginChrome(...)` 调用未包裹 `try/except`。若该方法抛出异常（Chrome 崩溃、网络超时、配置缺失），`code, msg, browser` 三个变量可能处于未赋值状态，导致 `UnboundLocalError`；异常还会直接中断整个 ETL 任务。
- **影响**: 高。任务非预期中断，且错误日志无法准确反映根因。
- **修复方案**:
  ```python
  def _login_with_retry(self, chromeUserPath):
      username = self.config['oc']['username']
      password = self.admin.decryptFernet(self.config['oc']['password'])
      last_msg = ''
      browser = None
      for attempt in range(self.api_retry):
          try:
              code, msg, browser = self.service.loginChrome(chromeUserPath, username, password)
              if code == 0:
                  return 0, msg, browser
              last_msg = msg
              self.logger.warning(f"登录橙联失败(attempt {attempt + 1}/{self.api_retry})，{msg[:200]}")
              if 'user data directory' in msg.lower() or 'invalid argument' in msg.lower():
                  BrowserUtils.clear_error_chrome()
                  time.sleep(3)
              elif attempt < self.api_retry - 1:
                  time.sleep(self.api_retry_interval)
          except Exception as e:
              last_msg = str(e)
              self.logger.warning(f"登录橙联异常(attempt {attempt + 1}/{self.api_retry})，{last_msg[:200]}")
              if attempt < self.api_retry - 1:
                  time.sleep(self.api_retry_interval)
      return -1, last_msg, browser
  ```

### 🟠 严重（推荐修复）

#### 2. `res.get_msg()[:N]` 存在 TypeError 风险
- **位置**:
  - `OcInv.py:51` `res.get_msg()[:200]`
  - `OcOrder.py:87` `res.get_msg()[:200]`
  - `OcOrder.py:110` `postcode_res.get_msg()[:200]`
  - `OcOrder.py:158, 166, 177, 185` 等重试日志
- **问题**: 若 `ResultObj.get_msg()` 返回 `None`，切片操作将抛出 `TypeError: 'NoneType' object is not subscriptable`。虽然 `ResultObj` 通常返回字符串，但无类型保证，且外部系统错误可能导致异常路径返回 `None`。
- **影响**: 中。在原本已失败的路径上二次抛异常，掩盖原始错误。
- **修复方案**: 统一封装安全切片方法，或每次做防御性处理：
  ```python
  msg = (res.get_msg() or '')[:200]
  ```

#### 3. 重复重试逻辑未抽象，维护成本高
- **位置**:
  - `OcInv.py:132-154` (`jsapi_purchase_history`)
  - `OcOrder.py:148-170` (`jsapi_outbound_order_trac_detail`)
  - `OcOrder.py:172-194` (`jsapi_outbound_order_postcode`)
- **问题**: 三个方法的重试逻辑几乎完全一致（try → 调用 → 判断成功 → 记录 last_error → sleep → 循环）。若后续需要调整重试策略（如指数退避、不同错误码不同处理），需要修改三处。
- **影响**: 中。代码重复，维护困难，容易遗漏。
- **修复方案**: 提取通用方法：
  ```python
  def _jsapi_with_retry(self, browser, url, param, action_name, extra_log=''):
      last_error = ''
      for attempt in range(self.api_retry):
          try:
              api_res = BrowserUtils.jsapi(browser, url, param)
              if api_res.is_success():
                  return ResultObj.success(json.loads(api_res.get_data()))
              last_error = api_res.get_msg() or ''
              self.logger.warning(f"{action_name}失败(attempt {attempt + 1}/{self.api_retry}){extra_log}，{last_error[:150]}")
          except Exception as e:
              last_error = str(e)
              self.logger.warning(f"{action_name}异常(attempt {attempt + 1}/{self.api_retry}){extra_log}，{last_error[:150]}")
          if attempt < self.api_retry - 1:
              time.sleep(self.api_retry_interval)
      return ResultObj.error(ResultObj.EXT_SYS_ERROR,
          f"{action_name}失败（{self.api_retry}次重试后）{extra_log}，最后错误：{last_error[:200]}")
  ```

### 🟡 一般（建议修复）

#### 4. `empty_retry` 计数逻辑行为变更，区域采集耗时可能增加
- **位置**: `OcInv.py:42-68`
- **问题**: 原逻辑中 `retry_count` 在 `data` 为空与 `data_list` 为空之间**共享**，新逻辑中两者**独立**（`empty_retry` 在 `data` 不为空时重置为 0）。这意味着：
  - 原逻辑：最多 3 次空重试（data 空 + data_list 空合计）
  - 新逻辑：最多 6 次空重试（data 空最多 3 次，data_list 空最多 3 次）
- **影响**: 低-中。若某区域长期返回空数据或异常 list，任务在该区域的停留时间可能翻倍。通常有利于稳定性，但需关注调度超时。
- **修复方案**: 确认此为预期行为，或统一 `empty_retry` 为整区域级计数器。

#### 5. 浏览器对象泄漏风险
- **位置**: `OcInv.py:112-130` (`_login_with_retry`)
- **问题**: 重试前调用 `BrowserUtils.clear_error_chrome()`，但未显式关闭旧 `browser` 实例。若 `loginChrome` 返回了一个已启动但登录失败的 Chrome 进程，可能导致僵尸进程累积。
- **影响**: 低。依赖 `clear_error_chrome()` 内部实现，若已做清理则无风险。
- **修复方案**: 在 `clear_error_chrome()` 前增加 `if browser: browser.quit()`（或相应关闭方法）。

#### 6. 缺少测试覆盖
- **位置**: 整个 PR
- **问题**: 未新增单元测试或集成测试。重试逻辑、跳过策略、失败统计等核心改动无自动化验证。
- **影响**: 中。回归风险高，后续重构缺乏安全网。
- **修复方案**: 至少为 `_login_with_retry` 和 `_jsapi_with_retry`（若提取）编写 mock 测试；或补充 ETL 集成测试用例。
