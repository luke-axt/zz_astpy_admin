## 变更概览

本次 PR 实现「通过企业微信远程触发 astpy 批处理任务」的完整链路，核心包含：

1. **企微文档 API 客户端** (`CoreBussiness/qywx/WeComDocTool.py`)：通过内部 astdc 代理调用 wedoc 接口，支持读表格、建文档、查信息。
2. **仓库参数同步业务** (`rpa/wecom/PlanAstdcDcWarehouseDoc.py`)：jobid=307，读取企微在线表格并同步到 `dc_warehouse`。
3. **远程执行 HTTP 网关** (`task/job_api_service.py`)：FastAPI 服务，提供 `/jobs/run` 接口，支持 Token 鉴权、并发控制、子进程隔离执行。
4. **子进程 Worker** (`task/run_job_worker.py`)：从 stdin 接收参数，执行 `JobCore.run_job`，stdout 返回结果。
5. **启动脚本与文档**：PowerShell/BAT 启动脚本及核心逻辑说明文档。

## 审查发现

### 🔴 高危（必须修复）

#### 1. `run_job_in_process` 缺少子进程超时控制，慢任务可导致服务 DOS

- **位置**：`task/job_api_service.py`，`run_job_in_process` 方法
- **问题**：`asyncio.create_subprocess_exec` + `proc.communicate()` **未设置 `timeout`**。若批处理任务死循环、死锁或执行数小时，子进程永不退出，HTTP 请求持续挂起，并发槽位被永久占用。
- **影响**：单个恶意或异常任务即可耗尽全部并发配额（默认 5 个），导致后续所有远程执行请求被拒绝（503），服务可用性严重受损。
- **修复方案**：
  ```python
  try:
      stdout_bytes, stderr_bytes = await asyncio.wait_for(
          proc.communicate(worker_payload.encode("utf-8")),
          timeout=self.env_int("ASTPY_JOB_SERVICE_TIMEOUT", 300),
      )
  except asyncio.TimeoutError:
      proc.kill()
      await proc.wait()
      return ResultObj.error(
          ResultObj.FATAL_ERROR,
          f"任务执行超时（>{self.env_int('ASTPY_JOB_SERVICE_TIMEOUT', 300)}秒），已强制终止",
      )
  ```

#### 2. `recerve_email` 拼写疑似错误，需确认数据库列名

- **位置**：`rpa/wecom/PlanAstdcDcWarehouseDoc.py:20`
- **问题**：`FIXED_SHEET_COLUMNS`、`FIELD_LABELS`、`UPDATE_FIELDS` 中均使用 `"recerve_email"`，正常英文拼写应为 `"receive_email"`。代码注释写的是"收货邮箱"，语义上应是 `receive`。
- **影响**：
  - 若数据库列名实际为 `receive_email` → 该字段**永远无法被正确映射和更新**，属于功能缺陷。
  - 若数据库列名确实是 `recerve_email`（历史遗留）→ 无功能影响，但属于技术债务。
- **修复方案**：立即确认 `dc_warehouse` 表实际列名。若实际为 `receive_email`，请在本次 PR 中统一修正；若确认为 `recerve_email`，请在代码注释中明确标注"历史遗留拼写，与数据库保持一致"。

### 🟠 严重（推荐修复）

#### 3. `/jobs/run` 未在网关层校验 jobid 有效性

- **位置**：`task/job_api_service.py`，`build_job_kwargs` / `handle_run_job`
- **问题**：仅校验 `jobid > 0`，未查询 `dc_job_info.is_valid='Y'` 确认该 jobid 是否允许远程执行。`list_valid_jobs` 接口列出了所有有效任务，但执行入口没有利用这份白名单。
- **影响**：可能执行已禁用（`is_valid='N'`）或不存在 jobid 的任务。虽然 `JobCore.run_job` 内部可能有兜底，但网关层作为安全边界应当提前拦截。
- **修复方案**：在 `handle_run_job` 的鉴权通过后、执行前，查询 `dc_job_info` 确认 jobid 有效：
  ```python
  # 伪代码
  if not self.is_job_valid(jobid):
      return self.to_http_response(
          ResultObj.error(ResultObj.PERMISSION_DENIED, "jobid 不存在或已被禁用"),
          request_id, http_status=403, jobid=jobid,
      )
  ```

#### 4. `run_job_in_process` 未处理子进程异常退出码

- **位置**：`task/job_api_service.py`，`run_job_in_process` 方法
- **问题**：处理了 stdout 为空、stdout 非 JSON、stderr 非空三种情况，但**未检查 `proc.returncode`**。若子进程被系统 OOM Killer 终止、段错误（SIGSEGV）或收到其他信号，`returncode` 非 0，但 stdout 可能仍有内容（或不完整）。
- **影响**：无法区分「业务失败」与「进程级异常」，排查困难。
- **修复方案**：在解析 stdout 后，若 `proc.returncode != 0`（Windows 下为负数），增加日志并调整返回信息：
  ```python
  if proc.returncode != 0:
      self.logger.error("子进程异常退出 returncode=%s", proc.returncode)
      # 可酌情将 returncode 附加到返回 data 中
  ```

#### 5. SQL 列名拼接（白名单控制下风险较低，但不符合安全规范）

- **位置**：`rpa/wecom/PlanAstdcDcWarehouseDoc.py`，`select_dc_warehouse_row` 与 `update_dc_warehouse_row`
- **问题**：使用 f-string 拼接列名到 SQL 语句：
  ```python
  col_clause = ", ".join(columns)
  sql = f"select {col_clause} from dc_warehouse where warehouse_name = %s"
  ```
- **影响**：当前 `columns` 来自硬编码白名单 `UPDATE_FIELDS`，注入风险可控。但若未来白名单被动态扩展（如从配置读取），将直接引入 SQL 注入漏洞。
- **修复方案**：在方法注释中显式声明"列名来自硬编码白名单 UPDATE_FIELDS，禁止从外部输入直接传入"；或改用查询构建器/ORM。

### 🟡 一般（建议修复）

#### 6. `values_equal` 中 `int()` 转换未捕获异常

- **位置**：`rpa/wecom/PlanAstdcDcWarehouseDoc.py`，`values_equal` 方法
- **问题**：`int(old_value) == int(new_value)` 未处理 `ValueError`。虽然 `new_value` 已走 `convert_value` 校验，但 `old_value` 来自数据库，极端情况下可能为非预期类型。
- **影响**：单个字段的类型异常将导致整个仓库更新失败。
- **修复方案**：
  ```python
  try:
      return int(old_value) == int(new_value)
  except (ValueError, TypeError):
      return False
  ```

#### 7. `read_json_payload` 异常信息丢失

- **位置**：`task/job_api_service.py`，`read_json_payload` 方法
- **问题**：两个 `except` 块返回完全相同的错误消息，第二个 `except Exception` 未记录原始异常。
- **影响**：线上出现请求体解析失败时，无法区分是 JSON 语法错误还是网络/编码问题。
- **修复方案**：至少记录 `logger.warning(..., exc_info=True)`，或返回带原始异常类型的消息。

#### 8. BAT 启动脚本解析 env 文件时对特殊字符敏感

- **位置**：`task/启动定时任务&启动远程调度服务.bat`

- **影响**：如果 `ASTPY_JOB_SERVICE_TOKEN` 或数据库连接字符串包含特殊符号，环境变量加载可能不正确。
- **修复方案**：在 `job-service.env` 模板或部署文档中明确标注"值中避免使用 `& | < >` 等特殊字符"；或改用 PowerShell 脚本作为推荐启动方式。

## 其他观察

### 设计亮点（值得肯定）

1. **进程隔离**：每个批处理任务在独立 Python 子进程中执行，避免长任务拖垮 HTTP 服务主进程。
2. **先校验后写入**：`PlanAstdcDcWarehouseDoc` 采用"全量校验通过后才执行更新"的策略，有效避免半成功脏数据。
3. **防时序攻击**：Token 比对使用 `hmac.compare_digest`，符合安全编码规范。
4. **并发控制**：使用 `threading.Semaphore` 限制同时执行的子进程数量，防止资源耗尽。
5. **request_id 全链路透传**：从 HTTP 网关到子进程 Worker 均携带 `request_id`，便于日志串联和问题定位。
6. **企微 API 兼容性处理**：`WeComDocTool` 对返回数据的 `data.result` / `result` / `grid_data` 等多套结构做了防御性兼容。

### 架构建议（不在本次 PR 阻塞范围）

- **长任务队列化**：当前并发控制是内存级 Semaphore，如果服务重启，正在执行的任务状态丢失。后续可考虑引入 Redis/RabbitMQ 等持久化队列，支持任务排队、重试、结果轮询。
- **输入参数 Schema 校验**：`/jobs/run` 的 `params` 完全透传，建议在网关层或 `JobCore` 层增加 Pydantic Schema 校验，防止非法参数导致深层业务异常。
