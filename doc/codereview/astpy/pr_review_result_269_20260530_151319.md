## 🟠 严重（推荐修复）

### 5. `AstdcService.send_qw_msg` 中 `res.get_data` 缺少调用括号 `()`  ok
- **位置：** `CoreBussiness/AstdcService.py:233`
- **问题：**
  ```python
  return ResultObj.error(ResultObj.EXT_SYS_ERROR,'发送企微消息异常',res.get_data)
  ```
  此处 `res.get_data` 后面缺少 `()`，实际传递的是 **bound method 对象** 而非企微返回的数据。
- **影响：** 当企微 API 返回 `errcode != 0` 时，错误 `ResultObj` 中封装的 `data` 是方法对象（如 `<bound method ResultObj.get_data of ...>`），而非实际的企微响应内容。运维人员排查问题时无法看到原始错误信息，严重影响故障定位效率。若下游对 `data` 做 JSON 序列化，还可能引发额外的序列化异常。
- **修复方案：** 添加括号：
  ```python
  return ResultObj.error(ResultObj.EXT_SYS_ERROR,'发送企微消息异常',res.get_data())
  ```

### 6. `AdlEbaySuspectOrderMonitorJob.send_qywx_message` 异常时仍不会重试  ok
- **位置：** `dwd/adl/ebay/AdlEbaySuspectOrderMonitorJob.py:131-148`
- **问题：** `except Exception` 块中现在有 `time.sleep(retry_interval)`，但仍然**缺少 `continue`**。异常发生后执行完 sleep，会继续执行循环后面的 `return ResultObj.error(...)`，不会进入下一次循环。
- **影响：** 网络超时、DNS 解析失败等异常发生时，重试机制仍然失效，只尝试一次就直接失败返回。相比之下，API 返回业务错误（`res.is_fail()`）时反而会重试，**异常路径的重试逻辑反而比正常错误路径更弱**。
- **修复方案：** 在 `except Exception` 块末尾添加 `continue`：
  ```python
  except Exception:
      err_msg = f"未知报错：{traceback.format_exc()}"
      time.sleep(retry_interval)
      continue  # <-- 添加此行
  ```
