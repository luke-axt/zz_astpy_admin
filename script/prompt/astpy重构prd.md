# 整体架构
## 现状
我有一个python项目：astpy。技术栈：python+window+mysql/sqlite+selenium,包含如下功能：
1. 实时数据采集：通过rpa、api等方式从外部系统或者公司外购系统采集数据。
2. 离线数仓：以数据处理和指标计算为主，主要就是跑sql，做数据清洗和指标计算。
3. window桌面应用：基于pyside6的桌面应用，主要给用户下载报表，维护业务数据用的。
4. rpa模块：负责跨系统的重复性操作的自动化，或者支持erp无法实现的关键功能（比如现在的尾程，或者外贸业务订单同步等）。
5. 定时任务模块：所有数据采集、离线数仓、rpa等需要定时启动的任务，都由定时任务模块负责调度，主要是通过apschedule实现，目前已经实现的分布式运行、T+1跑批的事件依赖等。
6. 有些需要7*24小时处理的任务，我会单独拆分出来，独立部署在云服务器，使用sqlite数据库，跟我内网生产环境隔离开。（我内网是没有不间断电源的，mysql数据库也是在内网中）
7. 所有的定时任务都是部署在window环境的，方便做爬虫。
8. 我还有一套若依系统，这套系统目前主要是做web交互，以及API服务，固定ip绑定在若依应用服务上。
9. 整个项目相当于一个桌面应用+大量批处理任务，批处理任务已经超过200个。系统已经对接8个海外仓系统、2个erp、1个金蝶云星空系统，还集成了百度云、腾讯云、163邮件、企业微信等基础服务能力。

由于我们从0-1的时候架构设计不合理，现在打算重构一下项目。

我们现在不写代码，只讨论设计方案。

## 重构方案
### 一、总体原则
1. 单一入口，统一继承：所有模块（服务、批处理、客户端、RPA、ETL）都从 BaseComponent 继承。
2. 配置驱动： 支持开发/测试/生产环境切换，通过配置文件或环境变量控制。
3. 多数据库支持：可动态切换数据库用户、方言（SQLite/MySQL/PostgreSQL 等）。
4. 可插拔架构：RPA 等模块可独立打包部署，自带 SQLite 数据库。
5. 日志、监控、错误处理统一：基类封装通用功能，避免重复代码。
6. 我不需要数据库连接池模块，因为我的项目是主要是定时任务，不是在线服务，定时任务都是短平快的任务，处理完就结束，运行时间超过15分钟的代码，我们会主动优化数据库连接的使用。
7. 我不需要api和rpa灵活切换的架构设计，因为项目的200多个作业运行2年多，没有这种场景出现过，这种架构会导致我维护两套代码。
8. 我会保持一个任务一个文件一个类的原则，保持高内聚设计。
9. 新的代码目录规划参考如下的【目录规划】
10. 按照新的目录规划，core库用来沉淀通用逻辑和规则，任务类来处理特定流程的编排或数据整理，task类负责所有批处理任务的分布式定时调度。
11. 类设计：
采用分层和模块化设计基类，项目采用三级基类。第一级基类：全局统一基类，只读取基础配置和环境变量即可；这里应该能够识别是独立部署则不用读取环境变量，如果是集成部署则需要读取环境变量。
第二级基类：视图类；任务类；服务类；实体类；其中视图类用于pyside6应用，任务类用于rpa和etl等任务的基类，服务类用于外部系统接口，实体类用于数据库实体类或者业务对象实体。
第三级基类：根据实际应用场景和设计引入数据库用户，多数据库用户在这里实现。
12. 配置管理
代码的配置只用.py脚本，不要用yml，方便pyside6打包。
本地配置只记录数据库连接信息，数据库密码记录在环境变量中。其他配置记录在数据库中。如果是sqlite部署，配置默认保存在本地的ast_sys_config.db中。
根据服务器的实际情况，在D:\adata或者userhome：\adata作为应用的数据根目录，在数据根目录下的sqlitedb保存sqlite文件。


## 目录规划
astpy/
├── asrc/                        # [核心] 源代码根目录 (遵循标准Python布局)
│   ├── core/                     # [核心层] 全局共享基础设施 (解决utils和API集成问题)
│   │   ├── config/               # 全局配置管理 (数据库连接、系统开关)
│   │   ├── base/                 # 存放统一的基类
│   │   ├── database/             # 数据管理,对应现在的astpy\admin\service\DBcore.py
│   │   ├── utils/                # 一些基础的工具类方法，对应现在的 astpy\utils
│   │   ├── entity/               # 一些通用实体类，比如通用的基类，或者是一些已经设计好的业务对象类。对应现在的：astpy\common
│   │   └── service/              # [关键] 外部系统API集成 (ERP, 金蝶, 百度云等)，对应现在的astdc.service目录
│   │
│   ├── etl/                      # 数据采集和数据仓库，整合现在的dwd+ETL目录
│   ├── rpa/                      # RPA 自动化脚本 (调用 core.external 或 selenium)，对应原来的rpa目录
│   ├── client/                   # [应用层-交互] 桌面应用入口，对应原来的client目录，pyside6应用
│   │
│   ├── tasks/                    # [应用层-调度] 定时任务，基于数据库中的作业配置，实现批处理任务类的动态导入和分布式调度
│
├── tests/                        # 测试目录
├── scripts/                      # 运维脚本 (部署、数据库迁移)
├── pyproject.toml                # 项目依赖与配置
└── README.md
